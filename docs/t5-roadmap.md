# T5 整体规划

## T5 阶段定位
T5 是 V1 阶段的最后一个功能模块，目标是补齐“导出/导入数据 + 推送提醒”的辅助能力。  
完成后即进入 T6（视觉对齐）阶段。

## T5 视觉范围
T5 阶段所有新页面、新组件、新交互均遵循 T6 之前的简洁视觉约定，
不做视觉对齐 v1 原型。视觉打磨统一在 T6 阶段完成。

## 实现 T5 时必须遵守的前序契约

- 任何写血压记录的逻辑（含 T5.5 CSV 导入批量调 `saveRecord`）必须严格遵守 [t1-contracts.md](docs/t1-contracts.md) 的 `measuredAt Contract` 和 `Record Type Contract`
- 报告渲染（T5.1）涉及阈值判断时，使用 `profiles.settings.bp.threshold` 而非 `referenceLines`，参考 [t1-contracts.md](docs/t1-contracts.md) 的 `Blood Pressure Settings Contract`
- 推送相关云函数（T5.3 / T5.4）的权限校验沿用 [t4-contracts.md](docs/t4-contracts.md) 的角色权限矩阵，特别是 `subscribeAlerts` 字段的语义
- 任何新云函数的部署与本地验证遵循 [deployment-notes.md](docs/deployment-notes.md) 的三层 gate
- 写操作的缓存更新沿用 T2.5 SWR + T4.2b staleness 的混合策略

## T5 拆分
按以下顺序执行，每个子阶段独立 commit + 烟测：

### T5.1 就诊报告导出
- 入口：档案详情页“生成就诊报告”卡片（参考 v1 原型 `report-page.html`）
- 时间范围切换：7 / 30 / 90 天
- 报告内容：
  - 患者档案（姓名、年龄、长期用药、紧急联系人）
  - 异常警示（血压过高风险等，红色调）
  - 数据统计摘要（测量次数、均值、异常次数）
  - 血压波动趋势图
  - 心率变化趋势图
  - 最近异常明细（前 5 条）
  - 用药信息（前 3 条 active 用药，顿号连接）
  - 免责声明
- 隐藏敏感信息 toggle（导出图片中紧急联系人脱敏为 `***`）
- 导出：保存为图片到相册
- 权限挽回弹窗（无相册权限时引导用户开启）
- 角色权限：所有角色（owner / collaborator / viewer）都能查看和导出
- 工作量：4-6 天

### T5.2 单图表导出
- 档案详情页的趋势图右上角加 `⬇` 图标
- 点击保存当前图表为图片
- 复用 T5.1 的 canvas 渲染基础设施
- 工作量：1.5-2 天

### T5.3 异常推送
- 订阅消息授权（在录入血压成功后或档案详情页适当时机请求）
- 异常推送触发：`saveRecord` 云函数检测到 `alertTriggered` 时，向所有 `subscribeAlerts = true` 的成员推送
- 推送内容：模板消息（具体模板需要在微信公众平台申请）
- T5.3 阶段还要补“测压提醒”基础设施的一部分
- 工作量：5-7 天

### T5.4 测压提醒
- 用户级订阅（在用户设置页加“测压提醒”区）
- 早晨 / 睡前提醒，时间可设
- 通过定时云函数 + 订阅消息推送
- 工作量：3-4 天

### T5.5 数据导入导出（简化版）
- 全部记录页右上角加 `⬇` 图标
- 导出：CSV 文本（标准格式：`日期,时间,高压,低压,心率`）
- 导入：粘贴 CSV 文本，前端解析后批量调 `saveRecord`
- 不做 OCR 图片识别
- 不做“按市面血压软件标准”——就用我们自己的简单 CSV 格式
- 工作量：3-4 天

## T5 阶段总工作量预估
约 17-23 天，约 3.5-5 周。

## T5 完成后的状态
V1 功能完整。进入 T6（视觉对齐 24 个原型）。

## 跨子阶段共享的基础设施
- canvas 长图渲染（T5.1 建立，T5.2 复用）
- 订阅消息授权（T5.3 建立，T5.4 复用）
- CSV 解析与生成（T5.5 单独）

## 不做的事（推到 V1.1+）
- OCR 图片识别（数据导入）
- 报告内容用户配置 toggle（除“隐藏敏感信息”外）
- 全部记录数据表导出为图片（场景 2，原本想做但产品价值不明确）
- 多种“市面标准”格式适配（欧姆龙 / 血压宝等）
