## Context

当前档案删除链路只有 `cloudfunctions/deleteProfile/handler.js`，行为是 owner 将 `profiles.deletedAt` 写为当前时间。`auth.getProfile(..., { includeDeleted: false })` 会把软删除档案从正常业务路径中隐藏，因此数据页、档案页、记录与用药查询都会自然看不到该档案。

现状缺口有两个：
- 软删除后没有恢复入口，误删只能靠人工改库。
- 软删除后没有最终清理，`relationships`、`records`、`medications` 会无限期保留。

这次变更跨越云函数、数据生命周期和运维调度，属于需要先定清楚生命周期语义再编码的改动。

## Goals / Non-Goals

**Goals:**
- 给档案删除定义明确生命周期：删除、保留、恢复、清理。
- 在不破坏现有“软删除后立即隐藏”的前提下，增加 30 天内可恢复能力。
- 在超过 30 天后，对 `profiles`、`relationships`、`records`、`medications` 做定时物理清理。
- 让恢复与清理都具备可预期的权限与幂等语义。

**Non-Goals:**
- 不在这次设计里定义前端最终 UI 文案和交互动效。
- 不扩展到 `invitations`、导出图片或其他非档案主数据表的清理策略。
- 不改变现有单条记录、单条用药自己的软删除行为。
- 不处理“删除后 30 天内自动通知用户即将清理”等提醒机制。

## Decisions

### 1. 继续以 `profiles.deletedAt` 作为删除生命周期唯一来源

保留现有软删除模型，不新增 `purgeAt`、`status` 一类字段。恢复窗口与清理截止时间统一按 `deletedAt + 30 天` 推导。

原因：
- 当前代码已经广泛依赖 `deletedAt` 隐藏档案，继续沿用改动最小。
- 30 天保留窗口是固定规则，不需要单独落库存一份冗余时间。
- 避免后续出现 `deletedAt` 与 `purgeAt` 不一致的双字段状态机。

备选方案：
- 新增 `purgeAfterAt` 字段，方便定时任务直接查询。
- 不采用，因为现阶段查询条件 `deletedAt <= cutoff` 已足够表达需求。

### 2. 新增独立 `restoreProfile` 云函数，权限仅限 owner

恢复不复用 `updateProfile`，而是走独立 `restoreProfile` 云函数：
- 调用者必须是该档案 owner。
- 档案必须存在且当前处于软删除状态。
- 若 `deletedAt` 已超过 30 天窗口，则返回不可恢复错误。
- 成功时将 `profiles.deletedAt` 清空，并更新 `updatedAt`。

原因：
- “恢复”是明确的生命周期动作，单独入口比 patch 型更新更容易做权限和错误码控制。
- 现有 `updateProfile` 语义是编辑档案资料，不适合承载生命周期恢复。

备选方案：
- 在 `updateProfile` 增加 `restore: true` 分支。
- 不采用，因为会把普通资料更新和生命周期操作耦合在一起。

### 3. 物理清理只以 `profiles` 为主驱动，级联硬删四张表

定时清理任务按批扫描超过 30 天的软删除档案，然后按 `profileId` 级联删除：
- `relationships`
- `records`
- `medications`
- 最后删除 `profiles`

原因：
- `profiles` 是档案生命周期主表，其他三张表都是从属数据。
- 只要 `profile` 仍存在，就保留恢复可能；真正超期后再统一物理删从属数据。
- 级联范围和用户要求严格一致，不把其他表误纳入首版清理。

备选方案：
- 删除时就同步软删或硬删四张表。
- 不采用，因为会破坏 30 天内可恢复的语义。

### 4. 清理任务使用批处理 + 幂等设计，不要求跨四表大事务

新增定时云函数（如 `cleanupDeletedProfiles`）：
- 每次按固定批次拉取符合条件的 profile。
- 对每个 profile 分步骤删除从属表，再删除 profile 本身。
- 若中途失败，下一轮任务可重试，已删数据不要求回滚。

原因：
- 四张表的数据量可能较大，跨集合大事务容易受云开发事务和批量限制约束。
- 清理任务天然适合幂等重试，追求“最终一致”比强事务更现实。

备选方案：
- 单 profile 整体事务删除。
- 不采用，因为记录/用药数量上来后很容易超出事务能力。

### 5. 删除后的业务可见性保持现状，恢复后重新参与正常查询

删除后无需额外给 `relationships` 打状态字段。因为当前所有需要读取档案的主路径都经过 `auth.getProfile` 或 `profiles.deletedAt: null` 过滤，软删除档案已经不会出现在正常列表里。恢复后只需清掉 `deletedAt`，原有关系、记录、用药即可重新生效。

原因：
- 这是最小侵入方案，避免为 `relationships` 增加第二套隐藏逻辑。
- 也符合“保留期内是隐藏但可恢复”的产品语义。

## Risks / Trade-offs

- [清理中途失败导致半清理] → 通过幂等任务设计和下次重试兜底；实现时输出按 profile 粒度的清理日志。
- [30 天窗口边界理解不一致] → 统一按绝对时间 `deletedAt + 30*24h` 判定，不按自然月。
- [恢复入口缺少前端承接] → 规格里先定义后端能力与语义，前端实现任务里补删除后入口和状态提示。
- [误删后成员关系被长期保留] → 这是保留恢复能力的必要代价；超过 30 天后统一物理清理。

## Migration Plan

1. 保持现有 `deleteProfile` 软删除语义不变。
2. 新增 `restoreProfile` 云函数和对应前端恢复入口。
3. 新增定时清理云函数，并先在开发/测试环境用人工触发验证批处理逻辑。
4. 配置线上定时触发。
5. 对历史已软删除档案按同一规则纳入清理：`deletedAt` 超过 30 天者会在任务上线后被清理。

回滚策略：
- 若恢复入口或清理任务出现问题，可先停用定时任务；现有软删除行为仍可继续工作。
- 已被物理清理的数据不可回滚，因此上线前必须先完成测试环境验证。

## Open Questions

- 恢复入口最终放在档案页、设置页还是独立“回收站”入口，仍需产品确认。
- 定时清理任务是否需要额外告警/报表输出，当前先按基础日志设计。
